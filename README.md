# InkScope｜可视化手写 Agent 画布

InkScope 是一个基于手写交互的 AI Canvas 原型系统。  
它尝试将传统 Chat 类 AI 的线性对话方式，转化为一个可以手写、选择、连接、派生、剪枝和管理上下文的可视化思考空间。

用户可以像在 iPad 白板上做笔记一样，在画布中写下想法、圈选内容、调用不同 Agent 进行分析，并将 AI 生成结果转化为可继续操作的节点卡片。节点之间通过箭头建立上下文关系，使 AI 的生成过程不再只是一次性回答，而是可以被用户持续组织、追踪和再加工的思考网络。

## 项目背景

在传统 Chat 类 AI 工具中，用户通常通过文字输入问题，AI 以线性对话形式给出回答。这种方式虽然高效，但在复杂思考任务中存在一些问题：

1. **上下文难以可视化**  
   用户很难直观看到某个结论来自哪里，也很难追踪多个回答之间的关系。

2. **生成内容难以组织**  
   AI 输出通常堆叠在聊天窗口中，用户需要手动复制、整理、归类。

3. **发散思考不够自然**  
   设计、产品、研究类任务往往需要同时展开多个方向，而线性聊天不适合表达分支结构。

4. **用户对 AI 过程的控制感不足**  
   用户很难主动决定哪些内容保留、哪些内容剪枝、哪些内容继续派生。

InkScope 的设计目标是：  
让用户像在白板上思考一样，与 AI 进行可视化协作。

---

## 项目定位

InkScope 不是一个普通聊天机器人，而是一个面向设计思考、产品分析、知识整理和研究发散的手写式 AI 协作画布。

它的核心价值在于：

- 用手写降低输入门槛；
- 用套索选择替代复杂指令输入；
- 用 Agent 角色拆分不同思考视角；
- 用节点卡片承载生成内容；
- 用箭头关系保留上下文路径；
- 用画布空间支持发散、整理和二次生成。

---

## 核心理念

InkScope 关注的不是“让 AI 替用户完成思考”，而是让 AI 的参与过程变得：

- **可见**：生成内容以节点形式出现在画布中；
- **可控**：用户可以选择、连接、删除、裁剪和再次生成；
- **可追踪**：节点之间通过箭头记录上下文关系；
- **可组织**：用户可以将不同 Agent 的结果放入同一个思考空间；
- **可迭代**：生成内容不是终点，而是下一轮思考的素材。

## 核心功能

### 1. 手写输入

用户可以使用钢笔工具在画布上书写想法、关键词、问题或草稿。

适合输入：

- 产品问题
- 设计概念
- 用户需求
- 研究关键词
- 头脑风暴内容
- 初步方案想法

---

### 2. 套索选择

用户可以使用套索工具选择手写内容。  
系统会识别选区中的手写信息，并将其作为上下文传递给 Agent。

这种交互方式避免了用户必须反复打字输入问题，使 AI 调用更接近真实白板工作流。

---

### 3. 多 Agent 调用

InkScope 支持多个不同角色的 Agent。  
每个 Agent 代表一种不同的思考视角。

默认 Agent 包括：

| Agent | 作用 |
|---|---|
| 助理 | 总结、解释、辅助整理 |
| 技术顾问 | 分析技术路径、实现方案和风险 |
| 执行经理 | 拆解任务、制定计划和行动步骤 |
| 知识顾问 | 扩展概念、补充知识和结构化信息 |
| 产品经理 | 分析用户需求、功能价值和优先级 |

用户可以根据当前任务选择不同 Agent 介入，而不是让单一 AI 角色回答所有问题。

---

### 4. 节点卡片生成

AI 生成的内容会自动转化为节点卡片。  
每个节点卡片包含：

- Agent 名称；
- 生成内容；
- 来源上下文；
- 与其他节点的关系；
- 可继续操作的内容区域。

节点不是静态结果，而是画布中的可操作对象。

---

### 5. 二次生成

用户可以继续选择已有节点中的内容，再调用其他 Agent 进行二次生成。

例如：

```text
用户手写：MVP 是什么？
↓
知识顾问：解释 MVP 的定义和核心特征
↓
产品经理：分析 MVP 对当前产品的功能优先级
↓
技术顾问：判断 MVP 方案的实现难度
↓
执行经理：拆解 MVP 的开发计划
```

通过这种方式，用户可以围绕一个问题逐步展开多视角思考。

---

### 6. 上下文箭头

节点之间可以通过箭头建立上下文关系。  
箭头表示内容之间的派生、引用、补充或合并关系。

这使用户能够清楚看到：

- 哪个内容来自哪个手写想法；
- 哪个 Agent 生成了哪个节点；
- 哪些节点被用于后续生成；
- AI 推理链路如何逐步展开。

---

### 7. 自由连接与裁剪

用户可以管理节点之间的关系：

- 新建箭头；
- 调整关系方向；
- 删除不需要的连接；
- 使用剪刀工具裁剪箭头；
- 保留有价值的上下文链路。

这让用户能够主动整理 AI 生成内容，而不是被动接受系统自动组织。

---

### 8. 画布操作

InkScope 支持基础白板操作：

- 画布拖动；
- 节点移动；
- 手写笔迹擦除；
- 节点删除；
- 箭头裁剪；
- 撤销操作；
- 设置面板；
- Agent 状态查看。

## 使用说明

### 第一步：进入系统

打开在线 Demo 链接后，会看到欢迎介绍界面。

点击：

```text
开始使用
```

进入 InkScope 画布。

---

### 第二步：在画布上手写

选择底部工具栏中的钢笔工具，在画布上写下想法。

示例：

```text
MVP 是什么？
```

也可以写：

```text
这个功能的用户价值是什么？
```

或者：

```text
如何优化这个产品的交互体验？
```

---

### 第三步：使用套索选择内容

选择套索工具，圈选刚刚手写的内容。

系统会将选区中的手写内容识别为文本，并准备调用 Agent。

---

### 第四步：选择 Agent

从左侧 Agent 栏中选择一个角色，例如：

```text
产品经理
```

系统会基于你选择的内容和 Agent 角色生成分析结果。

---

### 第五步：查看生成节点

AI 输出会生成一个新的节点卡片。  
节点卡片会出现在画布中，并与来源内容建立箭头连接。

---

### 第六步：继续二次生成

你可以继续选择节点中的一段文字，再调用另一个 Agent。

例如：

```text
知识顾问生成概念解释
↓
产品经理分析功能价值
↓
技术顾问判断实现路径
↓
执行经理拆解任务计划
```

这样可以形成一条清晰的 AI 协作思考链。

## 技术结构

InkScope 采用前后端一体的轻量结构：

```text
前端页面
↓
server.js 后端代理
↓
AI 模型 API
```

前端负责：

- 画布渲染；
- 手写交互；
- 工具栏操作；
- 节点与箭头展示；
- Agent 选择；
- 用户交互状态管理。

后端负责：

- 读取环境变量；
- 保护 API Key；
- 转发 AI 请求；
- 提供 `/api/chat`；
- 提供 `/api/ocr`；
- 提供 `/api/health`。

---

## API 安全设计

InkScope 不在前端暴露 API Key。

正确调用链路是：

```text
用户浏览器
↓
InkScope 前端
↓
server.js 后端接口
↓
豆包 / 火山方舟 API
```

这样可以避免用户在浏览器中看到真实 API Key。

## 本地运行

### 1. 克隆项目

```bash
git clone 你的 GitHub 仓库地址
cd handwriting-ai-canvas
```

---

### 2. 创建环境变量文件

复制 `.env.example`：

```bash
cp .env.example .env
```

Windows PowerShell：

```powershell
copy .env.example .env
```

---

### 3. 配置 `.env`

在 `.env` 中填写模型服务配置：

```env
DOUBAO_API_KEY=你的豆包APIKey
DOUBAO_BASE_URL=https://ark.cn-beijing.volces.com/api/v3
DOUBAO_CHAT_MODEL=你的接入点ID
DOUBAO_OCR_MODEL=你的接入点ID

DEFAULT_TEXT_PROVIDER=doubao
DEFAULT_OCR_PROVIDER=doubao

PORT=8078
```

注意：

```text
不要把 .env 上传到 GitHub。
不要在前端代码中写真实 API Key。
```

---

### 4. 启动服务

```bash
node server.js
```

本地访问：

```text
http://localhost:8078
```

---

## 设计亮点

### 1. 从 Chat 到 Canvas

InkScope 将 AI 交互从线性聊天转化为可视化画布，使用户可以在空间中组织想法。

### 2. 从输入指令到手写选择

用户不需要反复输入复杂 prompt，而是通过手写和套索选择自然触发 AI。

### 3. 从一次性回答到持续派生

AI 生成内容可以继续被选择、分析和扩展，形成连续思考链路。

### 4. 从黑箱生成到上下文可追踪

节点和箭头记录了生成来源，使 AI 协作过程更加透明。

### 5. 从单一 AI 到多 Agent 协作

不同 Agent 代表不同分析视角，帮助用户从产品、技术、知识和执行层面展开思考。

## 项目价值

InkScope 试图回答一个问题：

> 当 AI 不再只是聊天窗口，而是成为用户思考画布中的协作者时，交互方式应该如何变化？

它不是简单地把 AI 放进白板，而是尝试重新组织用户、内容、Agent 和上下文之间的关系。

在这个系统中：

- 用户负责提出问题和管理方向；
- Agent 负责从不同视角生成内容；
- 画布负责承载和组织思考过程；
- 箭头负责记录上下文关系；
- 节点负责沉淀可复用的信息。

这种方式更适合复杂问题的发散、比较、筛选和迭代。
